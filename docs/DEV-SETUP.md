# Dev Setup вЂ” PPCRV POC

## Overview

A local proof-of-concept for the PPCRV election results dashboard. The pipeline:

```
CSV (results + precincts)
  в†’ DuckDB aggregation (6 geographic levels)
  в†’ Partitioned Parquet files
  в†’ NestJS API (shells out to DuckDB CLI for on-read queries)
  в†’ Next.js UI (cascading geographic selection panel + results table)
```

## Technologies

### Monorepo вЂ” pnpm + Turborepo

- **pnpm workspaces** (`pnpm-workspace.yaml`) вЂ” manages `apps/*` and `packages/*` as a single repo
- **Turborepo** (`turbo.json`) вЂ” parallel task execution with caching
  - `pnpm dev` в†’ runs `nest start --watch` (API) + `next dev` (web) concurrently
  - `pnpm build` в†’ builds API (tsc) + web (next build) with dependency ordering
  - `pnpm test` в†’ runs Jest (API) tests
- **`.npmrc`** вЂ” `shamefully-hoist=true` required by NestJS's decorator metadata reflection

#### Why pnpm + Turborepo
- Disk-efficient (hard links, not copies)
- Strict module isolation by default (`. npmrc` loosens it for NestJS compat)
- Turborepo's `persistent: true` keeps both dev servers running in parallel

### API вЂ” NestJS (TypeScript)

- **Framework:** NestJS 10 with Express platform
- **Validation:** `class-validator` + `class-transformer` DTOs + global `ValidationPipe`
- **Query Engine:** Shells out to `duckdb` CLI binary (`execSync`), passes SQL with `-json` flag
- **CORS:** Enabled globally (port 3000 в†’ 3001)
- **Test:** Jest + `@nestjs/testing` with `ts-jest`

#### Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/results?level=...&contest=...&reg=...` | Aggregated candidate results |
| GET | `/api/contests` | List all contest codes |
| GET | `/api/regions` | List all region names |
| GET | `/api/regions/:reg/provinces` | Provinces in a region |
| GET | `/api/regions/:reg/provinces/:prv/municipalities` | Municipalities in a province |
| GET | `/api/regions/:reg/provinces/:prv/municipalities/:mun/barangays` | Barangays in a municipality |
| GET | `/api/barangays/:brgy/voting-centers` | Voting centers in a barangay |

All cascading endpoints filter by parent context via DuckDB `WHERE` clauses.

### UI вЂ” Next.js 14 (React 18)

- **Framework:** Next.js 14 App Router
- **Styling:** Tailwind CSS 3 with custom PPCRV brand palette
- **State:** React `useState` + `useCallback` (no external state lib)
- **API URL:** `NEXT_PUBLIC_API_URL` env var (default `http://localhost:3001/api`)
- **Client-side rendering:** Results page is `'use client'` for interactive cascading dropdowns

### ETL вЂ” Python + DuckDB

- **Aggregation engine:** DuckDB (in-process OLAP, no server needed)
- **CLI tool:** `scripts/run_aggregation.py` вЂ” CLI wrapper with `--sample N` flag
- **Core logic:** `aggregate_all_levels()` in `src/etl/aggregator.py`
  - Reads `results.csv` + `precincts.csv` via `read_csv_auto`
  - Joins on `LPAD(precinct_code, 8)` for precinct geo hierarchy
  - Aggregates votes at 6 levels in a single DuckDB SQL pass
  - Writes Hive-partitioned Parquet (`PARTITION_BY (contest_code)`)
- **Two aggregation modes:**
  1. **Precinct-level** (`processor.py`): Raw `SUM(VOTES_AMOUNT)` grouped by precinct/candidate/party
  2. **Multi-level** (`aggregator.py`): Full geographic rollup (national в†’ region в†’ province в†’ municipality в†’ barangay в†’ precinct)
- **Testing:** pytest with 13 total tests across processor + aggregator

### Key: DuckDB CLI On-Read Query Pattern

The API does NOT import the DuckDB Python library. Instead:

```
NestJS в†’ execSync("duckdb -json -c \"SELECT ... FROM 'output/multi-level/national/**/*.parquet'\"")
        в†’ DuckDB CLI reads Parquet files directly
        в†’ Returns JSON to stdout
        в†’ NestJS parses JSON в†’ returns response
```

This avoids the overhead of a running database server. For production, swap to DuckDB Python bindings or a real database.

## Setup

### 1. Clone and install JS deps

```bash
git clone <repo-url> pprcv-poc
cd pprcv-poc
pnpm install
```

### 2. Install Python ETL deps

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
# or: pip install duckdb pytest pyarrow psycopg2-binary
```

### 3. Verify DuckDB CLI

```bash
duckdb --version
# v1.4.5
```

### 4. Generate multi-level Parquet data

```bash
python3 scripts/run_aggregation.py sample-csv/results.csv sample-csv/precincts.csv output/multi-level --sample 100000
```

Omit `--sample` for full 24M-row dataset (takes a few minutes).

## Key Design Decisions

1. **DuckDB CLI instead of in-process DuckDB in Node.js** вЂ” NestJS shells out because `duckdb` Node.js bindings have native compilation issues; CLI is simpler for a POC and reliably available.

2. **Parquet files instead of a running database** вЂ” Zero maintenance, instant startup, no connection pooling. DuckDB reads Parquet directly with columnar pushdown for sub-millisecond queries.

3. **Hive-partitioned directories by `contest_code`** вЂ” Enables partition pruning: queries for a single contest only scan the relevant `contest_code=XXX/` subdirectory, not the full dataset.

4. **Cascading dropdowns fetch on-demand** вЂ” Each level fetches only when its parent is selected (e.g., provinces load only when a region is picked). No preloading all 93K geographic entries.

5. **`shamefully-hoist=true` in `.npmrc`** вЂ” NestJS decorator metadata reflection (`emitDecoratorMetadata`) requires dependencies to be resolvable from the hoisted root; pnpm's strict isolation breaks this without the flag.

## Common Issues

| Symptom | Cause | Fix |
|---------|-------|-----|
| `duckdb: command not found` | DuckDB CLI not installed | Install DuckDB CLI and ensure it's on PATH |
| `No files found that match the pattern` | Wrong Parquet path or glob | Set `PARQUET_BASE_PATH` or run `run_aggregation.py` first |
| `Cannot find module @nestjs/core` | Missing hoisted deps | Check `.npmrc` has `shamefully-hoist=true`, re-run `pnpm install` |
| Port 3000/3001 in use | Previous dev server still running | `lsof -ti:3000 -ti:3001 \| xargs kill -9` |
